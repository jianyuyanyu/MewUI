# Contributing to MewUI

MewUI에 관심을 가져주셔서 감사합니다.

이 프로젝트는 아직 빠르게 변화하는 단계에 있으며, v1.0으로 가는 과정에서 API, 내부 구조, 동작 방식이 계속 조정될 수 있습니다. 그래서 어떤 변경은 바로 PR로 받을 수 있지만, 어떤 변경은 구현보다 먼저 방향 합의가 필요합니다.

## 먼저 확인해 주세요

기여를 시작하기 전에, 내가 하려는 변경이 아래 중 어디에 해당하는지 먼저 구분해 주세요.

### 1. 작은 버그 수정

다음과 같은 경우는 보통 바로 PR을 열 수 있습니다.

- 명확한 회귀(regression) 수정
- 현재 의도와 분명히 어긋나는 동작 수정
- 문서, 샘플, 오탈자 등의 작은 수정
- 설계 영향이 제한적인 국소 변경

### 2. 동작 변경

다음과 같은 경우는 먼저 **Issue 또는 Discussion**으로 방향을 확인해 주세요.

- 입력 처리 방식 변경
- 스크롤, 포커스, 레이아웃, 렌더링 등 기존 동작 변경
- 기본값이나 UX 기대치를 바꾸는 변경
- 한 사용 사례에서는 좋아 보여도 다른 사용 사례에는 영향을 줄 수 있는 변경

이런 변경은 구현 자체보다도 **어떤 동작이 프로젝트에 맞는가**를 먼저 정하는 것이 중요합니다.

### 3. 새로운 아키텍처, 추상화, public API

다음과 같은 경우는 **사전 합의 없이 PR을 열지 않는 것**을 원칙으로 합니다.

- 새로운 백엔드 추상화
- public enum, interface, extension point 추가
- 수명주기나 구성 방식에 영향을 주는 변경
- 장기적인 유지보수 부담을 늘리는 구조 변경

이런 변경은 코드량이 작아 보여도 프로젝트 방향, 유지보수 비용, 호환성 기대치에 큰 영향을 줄 수 있습니다.

## 권장 절차

1. 먼저 관련 Issue, PR, Discussion이 이미 있는지 확인합니다.
2. 작은 버그 수정이면 바로 PR을 열 수 있습니다.
3. 동작 변경이면 먼저 Issue 또는 Discussion으로 문제 정의와 방향을 정리합니다.
4. 새로운 아키텍처, 추상화, public API면 사전 합의 후 PR을 엽니다.
5. 하나의 PR은 하나의 목적에 집중해 주세요.

## 이슈 작성 원칙

이슈는 가능한 한 **제공된 양식(template)** 에 맞추어 작성해 주세요.

- 버그 보고는 버그 리포트 양식을 사용합니다.
- 동작 변경 제안, API 제안, 설계 제안은 제안 양식을 사용합니다.
- 하나의 이슈에는 하나의 주제만 담아 주세요.
- 이미 비슷한 이슈가 있다면 중복 이슈 대신 기존 스레드에 맥락을 추가해 주세요.

### 버그 리포트에 포함되어야 하는 내용

- 사용 환경(.NET 버전, OS, 사용 패키지 또는 샘플 등)
- 재현 절차
- 기대한 동작
- 실제 동작
- 가능하다면 최소 재현 코드, 스크린샷, 로그

### 제안 이슈에 포함되어야 하는 내용

- 해결하려는 문제
- 현재 방식이 왜 부족한지
- 이것이 동작 변경인지, API 변경인지, 구조 변경인지
- 예상되는 사용자 영향 범위
- 가능한 대안이나 트레이드오프
- 코어 변경이 꼭 필요한지 여부

## Discussion을 먼저 써야 하는 경우

다음과 같은 경우는 Issue보다 **Discussion**이 더 적합할 수 있습니다.

- 탐색적인 아이디어
- 방향성 검토
- 아직 문제 정의가 충분히 정리되지 않은 제안
- 구현보다 설계 대화가 먼저 필요한 경우

## Pull Request 작성 원칙

PR을 열 때는 다음을 지켜 주세요.

- 해결하려는 문제를 분명히 설명합니다.
- 관련 Issue 또는 Discussion을 연결합니다.
- 현재 동작과 변경 후 동작을 설명합니다.
- 관련 없는 리팩터링은 섞지 않습니다.
- 변경 범위를 가능한 작고 명확하게 유지합니다.
- 필요한 경우 문서나 샘플도 함께 업데이트합니다.

### 바로 PR을 열지 말아야 하는 경우

다음과 같은 경우는 먼저 합의가 필요합니다.

- 유지보수자가 먼저 Issue/Discussion을 요청한 경우
- 프로젝트 방향과 범위 적합성 판단이 필요한 경우
- API 표면이나 아키텍처에 영향이 있는 경우
- 구현 전에 의도된 동작 정의가 필요한 경우

## 유지보수자가 할 수 있는 일

프로젝트를 관리 가능한 상태로 유지하기 위해 유지보수자는 다음과 같은 조치를 할 수 있습니다.

- PR을 Issue 또는 Discussion으로 먼저 전환하도록 요청
- 현재 방향과 맞지 않는 제안을 종료
- 범위를 더 작게 나누도록 요청
- 나중 마일스톤으로 연기
- 이미 답변된 제안이나 반복적 요청을 정리

## 스타일과 범위

- 기존 코드 스타일과 저장소 구조를 따라 주세요.
- 넓은 리팩터링보다 최소 변경을 선호합니다.
- 지금 해결하려는 문제와 직접 관련 없는 변경은 함께 넣지 말아 주세요.

## 행동 기준

이 저장소에 참여하는 모든 사람은 [Code of Conduct](CODE_OF_CONDUCT.ko.md)를 따라야 합니다.
